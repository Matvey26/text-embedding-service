# text-embedding-service

FastAPI-сервис для получения текстовых эмбеддингов через REST API на основе модели [rubert-mini-frida](https://huggingface.co/sergeyzh/rubert-mini-frida).

## Старт

### Установка зависимостей

```bash
uv sync
```

### Запуск

```bash
fastapi dev
```

Сервер запустится на `http://127.0.0.1:8000`.

## API

1. GET /health - Проверка статуса сервиса и готовности модели.

```json
// Модель ещё загружается
{
  "status": "loading",
  "model": "rubert-mini-frida",
  "detail": "Model is still loading"
}

// Модель готова
{
  "status": "healthy",
  "model": "rubert-mini-frida",
  "embedding_dim": 312,
  "detail": "Model is loaded and working"
}
```

2. POST /embed - Получение эмбеддинга для текста.

```bash
curl -X POST http://127.0.0.1:8000/embed \
  -H "Content-Type: application/json" \
  -d '{"text": "Привет мир"}'
```

**Параметры:**

| Поле | Тип | Обязательный | Описание |
|------|-----|--------------|----------|
| `text` | string | Да | Текст для эмбеддинга (1–2048 символов) |
| `prefix` | string | Нет | Префикс-подсказка. По умолчанию: `categorize` |

**Доступные префиксы:**

- `search_query` — для поисковых запросов
- `search_document` — для поиска по документам
- `paraphrase` — для парафразирования
- `categorize` — для классификации
- `categorize_sentiment` — для сентимент-анализа
- `categorize_topic` — для тематической классификации
- `categorize_entailment` — для логического вывода

## Структура проекта

```
app/
├── main.py       # Инициализация FastAPI-приложения, lifespan, background-загрузка модели
├── model.py      # Загрузка модели, токенизация, pooling, инференс
└── endpoints.py  # API-роуты (/embed, /health)
```

| Файл | Назначение |
|------|------------|
| `main.py` | Создаёт FastAPI app, регистрирует роуты, загружает модель в фоне при старте |
| `model.py` | Функции `init()`, `tokenize()`, `predict()`, `pool()` |
| `endpoints.py` | Эндпоинты `/embed` и `/health` с валидацией и обработкой ошибок |

## Выбор фреймворка

Я уже работал с flask и aiohttp. Flask неплохой, но для асинхронности, валидации и автоматической генерации документации требует установки целой кучи дополнительных зависимостей. Aiohttp это низкоуровневая библиотека, подходит в основном для написания клиентов, писать на ней сервис достаточно сложно. Я сравнил FastAPI с Flask и Aiohttp и его преимущества мне понравились (в нём есть то, чего так не хватает в flask), а вкупе с желанием попробовать что-то новое, выбор пал именно на этот фреймворк. 

## Метрики и целевые пороги

### Latency P50 / P95 / P99

**Что показывает:** Распределение времени ответа от запроса до полной выдачи. P99 — худший случай, который видят 1% пользователей.

**Почему важна:** Клиенты inference-сервисов часто работают в цепочках (RAG, автодополнение) — медленный ответ ломает весь пайплайн.

**Целевые пороги:**

| Метрика | Порог   | Обоснование                            |
|---------|---------|----------------------------------------|
| P50     | < 200ms | Типичный ответ для small-модели на CPU |
| P95     | < 500ms | Приемлемо для синхронных клиентов      |
| P99     | < 2s    | Худший случай — не чаще 1% запросов    |

**Если порог нарушен:**
- Если высокий P50
  - профилировать `predict()`;
  - переход на GPU или более лёгкую модель;
  - собирать запросы в батчи - это снижает оверхед и общую нагрузку;
  - пытаться использовать все ядра процессора для инференса
- Если высокий P99 или резко выше P50
  - решать проблему холодного старта;
  - профилировать ОС (не хватило RAM, ОС пошла в файл подкачки);
  - профилировать датчики железа (перегрелся процессор, частоты сбросились)

### Throughput (RPS)

**Что показывает:** Сколько запросов в секунду обслуживает сервис.

**Почему важна:** Определяет ёмкость — сколько параллельных пользователей потянет одна реплика.

**Целевой порог:** 20–50 RPS на CPU для `rubert-mini`. Измерять под нагрузкой.

**Если порог нарушен:**
- Горизонтальное масштабирование
- Батчинг запросов
- Оптимизация токенизации / уменьшение max_length

### Model Inference Time

**Что показывает:** Чистое время внутри `model(**tokenized_inputs)` — без HTTP overhead.

**Почему важна:** Отделяет производительность модели от производительности кода.

**Целевой порог:** < 50ms на CPU.

**Действия при нарушении:**
- `model.eval()` + `torch.inference_mode()`
- Квантизация модели
- torch.compile()

### CPU / Memory Usage

**Что показывает:** Утилизация процессора и потребление RAM.

**Почему важна:** Модель загружается целиком в память. Если RAM заканчивается — OOM, если CPU на 100% — запросы queueятся.

**Целевые пороги:**

| Ресурс | Порог        |
|--------|--------------|
| CPU    | < 80%        |
| Memory | < 70–80% RAM |

**Действия при нарушении:**
- Если CPU 100%, то масштабирование, батчинг
- Если Memory растёт, то уменьшение батчей

### Active Requests

**Что показывает:** Количество запросов, обрабатываемых в данный момент (хендлер выполняется, inference идёт).

**Почему важна:** В отличие от RPS (скорость), эта метрика показывает текущую нагрузку. Если Active Requests ≈ max_workers, а latency растёт — система насыщена и готова к отказам.

**Целевой порог:** Зависит от числа workers. Ориентир: Active Requests < 80% от max_workers. Резкий всплеск — сигнал к масштабированию.

**Действия при нарушении:**
- Если отношение Active Requests к workers растёт, то горизонтальное масштабирование
- Если резкий рост без роста RPS, то чинить latency

### Обоснование

Выбраны метрики, которые отвечают на вопросы "как быстро", "как много" и "хватает ли ресурсов".

| Группа   | Метрики                                   | Зачем                                 |
|----------|-------------------------------------------|---------------------------------------|
| Скорость | Latency P50/P95/P99, Model inference time | Реальный UX клиента                   |
| Ёмкость  | Throughput, Active Requests               | Сколько потянет одна реплика          |
| Ресурсы  | CPU, Memory                               | Предупреждают об исчерпании до отказа |

И кратко, почему другие метрики я не стал брать:

| Метрика     | Почему не выбрана                                   |
|-------------|-----------------------------------------------------|
| Queue depth | Нет явной очереди (Celery, Kafka) — нечего замерять |
| Error rate  | Следует из latency; добавить позже                  |
| Uptime      | SLA-метрика для прода, не для MVP                   |
| GPU VRAM    | GPU нет                                             |
| Network I/O | Не станет узким местом при CPU-сервисе              |
